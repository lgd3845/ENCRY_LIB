# 国密保密文件库设计与测试报告

## 目标与实现范围

本项目实现了一个 C++20 原生 Linux 可执行程序 `encrylib`，支持：

- 初始化保密文件库。
- 将文件移入文件库，默认成功后删除源文件。
- 浏览文件库中文件列表。
- 将文件移出文件库，默认成功后删除库内条目。
- 删除库内文件条目。
- 使用 Release 构建下的基准程序测试加解密效率。

未直接引用 OpenSSL、Botan 等密码算法库。SM3、HMAC-SM3、PBKDF2-HMAC-SM3、SM4 分组算法和 SM4-CTR 模式均在本项目内实现。

## 文件格式

文件库目录包含：

- `vault.meta`：明文保存 KDF 参数和盐值，密文保存 32 字节主密钥。
- `manifest.bin`：加密认证后的清单，包含文件名、对象 ID、原始大小和加入时间。
- `objects/*.obj`：每个文件一个对象，保存被包装的文件密钥和加密认证后的文件内容。

加密封装统一采用：

- 加密：SM4-CTR。
- 认证：HMAC-SM3。
- 认证覆盖 domain、AAD、nonce、密文长度和密文。
- 解密前先验证 HMAC，认证失败不输出明文。

## 密钥策略

本实现选择“每个文件使用不同文件密钥”，而不是所有文件共用一个内容加密密钥。

具体层级如下：

- 用户口令 + 32 字节随机盐，经 PBKDF2-HMAC-SM3 派生口令保护密钥。
- 随机生成 32 字节主密钥，使用口令保护密钥加密认证后写入 `vault.meta`。
- 从主密钥派生清单加密密钥、清单认证密钥、文件密钥包装加密密钥、文件密钥包装认证密钥。
- 每个文件生成独立 32 字节文件密钥。
- 从文件密钥派生该文件的内容加密密钥和内容认证密钥。

选择每文件独立密钥的原因：

- 单个文件密钥泄漏时，影响范围限于该文件。
- 后续可以扩展为单文件换钥、细粒度访问控制或文件级审计。
- 避免长期使用同一个内容密钥处理所有文件数据。

代价是每个对象需要额外保存一份被包装的文件密钥，格式稍复杂，但该开销相对文件内容通常很小。

## 用户密钥存储与使用策略

用户口令本身不落盘。磁盘上只保存：

- KDF 迭代次数。
- 随机盐值。
- 被口令派生密钥加密认证的主密钥。

默认 PBKDF2 迭代次数为 120000，CLI 初始化时可用 `--iterations` 提高。测试中为缩短运行时间使用了 10000。

内存中密钥数据使用 `SecureBytes` 或固定数组保存，并在析构或使用后通过 `secure_zero` 显式覆写。SM4 轮密钥、PBKDF2 中间值、派生子密钥和包装/内容密钥也会在生命周期结束时清零。

抵抗能力分析：

- 抵抗离线篡改：攻击者修改 `vault.meta`、`manifest.bin` 或对象密文后，HMAC-SM3 验证失败。
- 抵抗离线口令猜测：盐值防止预计算表复用，PBKDF2 提高每次猜测成本；安全性仍依赖用户口令强度。
- 限制单文件泄漏影响：每个文件独立文件密钥，单个文件密钥泄漏不直接解密其他文件。
- 保护文件名和清单元数据：清单整体加密认证，未解锁时不能列出文件名。

限制：

- 如果主密钥或进程内存被高权限攻击者读取，整个文件库可被解密。
- 不提供硬件安全模块、内核密钥环或 TPM 绑定。
- 不防止完整目录回滚攻击；攻击者若保存旧版 `vault.meta`、`manifest.bin` 和对象文件并整体替换，本地离线格式无法独立发现。
- `--password` 和 `ENCRYLIB_PASSWORD` 便于自动化，但可能被 shell 历史、进程环境或同机用户观察；人工使用建议交互输入。

## 构建与运行

Release 构建命令：

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)
```

测试命令：

```bash
ctest --test-dir build --output-on-failure
```

命令行示例：

```bash
./build/encrylib init ./vault
./build/encrylib add ./vault ./a.txt
./build/encrylib list ./vault
./build/encrylib extract ./vault a.txt ./out
./build/encrylib remove ./vault a.txt
```

最终产物：

- `build/encrylib`：主程序，可直接运行，不需要安装。
- `build/encrylib_tests`：测试程序。
- `build/encrylib_bench`：加解密性能基准程序。

## 性能测试

测试环境：

- 日期：2026-06-10。
- 系统：Ubuntu 24。
- 编译器：g++ 13.3.0。
- 构建：`cmake -DCMAKE_BUILD_TYPE=Release`，库目标启用 `-O3`。
- 计时：`std::chrono::steady_clock` 高精度计时。

命令：

```bash
./build/encrylib_bench
```

本机结果：

| 数据大小 | 加密耗时 | 解密耗时 | 加密吞吐 | 解密吞吐 |
|---:|---:|---:|---:|---:|
| 1 MiB | 14.525 ms | 14.491 ms | 68.845 MiB/s | 69.007 MiB/s |
| 8 MiB | 115.754 ms | 116.151 ms | 69.112 MiB/s | 68.876 MiB/s |
| 32 MiB | 273.495 ms | 274.277 ms | 117.004 MiB/s | 116.670 MiB/s |
| 128 MiB | 951.424 ms | 961.742 ms | 134.535 MiB/s | 133.092 MiB/s |

已做的简单优化：

- SM4 轮函数使用编译期生成的 T 表，把 S-box 后的线性变换合并为 4 次查表和异或。
- SM4-CTR 对 16 MiB 以上缓冲区按计数器块切分，最多使用 8 个 worker 并行生成和异或密钥流。
- SM3 预计算 `Tj <<< (j mod 32)` 轮常量，并拆分前 16 轮和后 48 轮，减少热循环分支和重复旋转。
- HMAC-SM3 的 64 字节 pad 改为固定数组，避免构造 HMAC 时的小堆分配，并保留显式清零。
- 认证标签计算中的 AAD/密文长度字段改为栈上 8 字节数组，减少临时分配。
- SM4-CTR 热路径使用紧凑块加密辅助函数，避免每个 16 字节计数器块都走通用块接口的 36 字中间状态擦除。
- SM4-CTR 输出异或按 64 位块处理，尾部不足 8 字节再逐字节处理。
- SM3 压缩函数去掉 `W'` 临时数组，轮函数中直接计算 `W[j] ^ W[j+4]`，并避免无用零初始化。

小输入仍保持约 69 MiB/s，避免为 1 MiB 和 8 MiB 数据引入线程开销；32 MiB 和 128 MiB 大输入因 SM4-CTR 并行化明显提升。128 MiB 端到端吞吐从最初约 54 MiB/s 提升到约 133 MiB/s。当前大输入速度主要受串行 HMAC-SM3 认证处理和内存带宽共同影响。后续优化方向：

- 对 SM4 批量 CTR keystream 生成做循环展开。
- 减少加密封装中的临时分配。
- 增加大文件流式加解密，避免一次性读入整个文件。
- 在不违反“禁止直接引用密码算法库”的前提下，使用平台 SIMD intrinsic 优化 SM4/SM3 内核。
